# SpawnWP 0.3.13

This release makes creating a site much faster and cuts the disk usage that grew with
every creation.

- **Site creation no longer rebuilds the PHP image every time.** The image is reused
  and rebuilt only when actually needed: on the first use of a PHP version, after a
  SpawnWP update that changes the image, or when it is older than 7 days (then with a
  fresh base pull, so a newly released WordPress still arrives within days). Creating
  a site on an already-built PHP version now takes about **35 seconds**; the one-off
  first build per PHP version takes about **5 minutes**, and the Deploy page shows a
  prominent notice when a creation includes it (and a "image already built" note when
  it does not). `SPAWNWP_IMAGE_MAX_AGE_DAYS` tunes the refresh window and
  `SPAWNWP_REBUILD=1` forces a fresh build.
- **Docker build cache stays small.** The previous per-creation rebuild left several GB
  of BuildKit cache behind. Stale cache is now trimmed right after each build, and the
  weekly prune drops cache unused for 72 hours instead of a week. An upgrade migration
  updates the prune unit on existing installs.
- **The MariaDB tuning config now actually applies.** The stack mounted a config file
  that did not exist, so Docker silently created an empty directory in its place and
  every site's database ran untuned, preallocating ~160 MB of empty InnoDB/Aria files.
  New sites use a small development profile (`docker/mariadb/spawnwp.cnf`): roughly
  60 MB less disk and a noticeably lower memory footprint per database.

Existing sites are not modified: they keep their frozen per-site stack and current
behaviour. Recreate a site to pick up the new database profile. The first site created
after this update rebuilds the PHP image once per PHP version (to stamp it with the new
build fingerprint); subsequent creations reuse it.
