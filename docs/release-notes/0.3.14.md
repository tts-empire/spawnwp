# SpawnWP 0.3.14

This release puts the PHP image lifecycle in your hands and adds the classic per-site
PHP knobs.

- **New System tab.** See every shared PHP image with its size (~1.8 GB each), age and
  the sites using it, next to a Docker disk breakdown. Keeping an image means each
  deploy on that PHP version takes ~35 seconds; **Delete** (allowed only when no site
  uses the image, typed confirmation) frees the space at the price of a ~5-minute
  rebuild on the next deploy; **Refresh** rebuilds an image right now with the latest
  WordPress, with streamed progress.
- **No more scheduled rebuilds.** 0.3.13 refreshed an image automatically once it was
  7 days old, making an occasional deploy unexpectedly slow. That is gone: images are
  only flagged as *stale* (System badge + a notice during deploys) and refreshed when
  you decide.
- **Optional auto-delete of unused images.** A System setting — "auto-delete unused
  images after N days", default 0 = manual only. Images referenced by at least one
  site are never auto-deleted. A daily timer performs the check; an upgrade migration
  installs it on existing hosts.
- **Per-site PHP settings.** The Deploy form gains a collapsed advanced section for
  `memory_limit`, `upload_max_filesize`, `post_max_size`, `max_execution_time`,
  `max_input_vars`, `max_input_time` and `display_errors`; the same values are
  editable later from **Manage → ⚙️ PHP settings** (a ~2-second php restart, no image
  rebuild — they live in a per-site override file mounted into the container).
  Raising the upload sizes aligns the site and proxy nginx limits automatically, up
  to 512M.

Existing sites keep working unchanged; the per-site PHP settings need the new mount,
so they are available for sites created from 0.3.14 onward (recreate a site to use
them). Existing PHP images keep their fingerprint — no forced rebuild on update.
