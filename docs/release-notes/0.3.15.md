# SpawnWP 0.3.15

This release rebuilds the Deploy experience and introduces disposable sites.

- **Redesigned Deploy page.** A stepped flow — choose a blueprint, configure, create —
  with aligned fields and a launch bar that shows the destination URL, a live
  **expected-time estimate** and the Create button. The estimate is real: it checks
  whether the chosen PHP version's image is already built and only warns about the
  one-off ~5-minute build when it will actually happen. The PHP settings section is
  now a proper collapsible panel with per-field hints, a "modified" chip and a
  reset-to-defaults action.
- **Deploy times on blueprint cards.** Each card shows its approximate deploy time
  (~35 seconds; ~1–2 minutes for Development, which installs plugins from
  wordpress.org), plus a clear note that the first deploy on a PHP version not built
  yet adds a one-off image build of about 5 minutes.
- **Temporary sites.** Pick a lifetime at creation (1, 3, 7 or 30 days) and the site
  destroys itself when it expires — containers, volumes, files and routes, with no
  backups: they are disposable by design. An hourly check enforces expiry (installed
  by a migration on existing hosts); Manage shows a countdown badge and a ⏳ Lifetime
  action to extend a site or make it permanent.
- **Fix: System tab stuck on "Loading".** 0.3.14 shipped new pages without bumping
  the asset cache-busting version, so browsers kept executing the previous
  JavaScript. The version string is now tied to the SpawnWP release and enforced by
  a static test. One hard refresh (Ctrl/Cmd+Shift+R) may be needed once after this
  update; never again afterwards.
- **Fix: orphaned nginx blocks after destroy.** On hosts serving from
  `sites-enabled/default`, destroying a site left its nginx location blocks behind —
  harmless at first, but dangerous once internal ports get reused by new sites.
  `destroy-project.sh` now resolves the enabled vhost exactly like site creation does.
