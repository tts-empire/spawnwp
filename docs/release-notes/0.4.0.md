# SpawnWP 0.4.0

This release adds **content blueprints**: turn a WordPress site you have already
configured — your usual plugins, theme, settings and starting content — into a
reusable blueprint on your own SpawnWP server, then spawn new sites from it in
minutes.

## Capture a site as a blueprint

1. On the cockpit's **System** page, open **Template connections** and generate a
   pairing code (single-use, expires in 15 minutes).
2. Install the **SpawnWP Deploy plugin 0.2.0+** on the configured source site and
   paste the code in the new *Create a SpawnWP blueprint from this site* panel.
3. Choose what to capture — plugin files, theme files, media uploads and the
   database are all included by default — give the blueprint an id, name and
   description, optionally widen the allowed PHP versions (the default pins to the
   source site's version), and press **Create blueprint**.

The capture is uploaded in checksummed chunks over the same Ed25519-signed request
format the plugin already uses for site-to-site deploys. The server verifies the
archive checksum, hardens the ZIP (no traversal, no symlinks, bounded expansion) and
installs manifest and payload atomically: a failed or interrupted capture can never
leave a half-installed blueprint. The new blueprint appears under **Your blueprints**
on the **Deploy** page, with its payload size, a capture summary and an estimated spawn
time.

## What you should know

- **Users and passwords are never captured.** The database export excludes the
  `users` and `usermeta` tables; every spawned site keeps its own fresh admin
  account. Content authorship is reassigned to that admin.
- **The source URL never reaches the server.** The capture rewrites it to a fixed
  placeholder, which each spawned site replaces with its own URL.
- **The database capture contains real content.** Posts, pages and settings from the
  source site will appear in every site spawned from the blueprint — the plugin asks
  for explicit confirmation before including it.
- **Licensed plugins may need attention.** Plugins that are not from WordPress.org
  are listed in the blueprint manifest and flagged at capture time and on the Deploy
  card: sites spawned from the blueprint may require new license keys or
  re-activation for them.
- **Re-capturing is safe.** Using an existing blueprint id offers a replace flow; the
  old payload is kept until the new one is verified, then swapped atomically.
- Payloads are capped at **2 GiB**; if a capture exceeds it, exclude the media
  uploads and re-run.
- Content blueprints can be deleted (manifest and payload) from **System → Content
  blueprints**. Existing sites are not affected.

## Under the hood

- Blueprint manifest **schema v2**, accepted only from `/etc/spawnwp/blueprints.d/`,
  with payload archives under `/var/lib/spawnwp/blueprints/<id>/`. Schema v1
  blueprints and their behaviour are completely unchanged.
- New signature-authenticated, rate-limited ingest API under `/api/ingest/`; pairing
  codes and connection management stay behind the cockpit session (pairing
  generation requires recent authentication). Existing installations receive the
  nginx configuration for the new endpoint through an update migration.
- Applying a content blueprint imports the database through temporary tables and an
  atomic `RENAME TABLE` swap, preserving the fresh site's URL and admin, then
  rewrites the placeholder URL everywhere (GUIDs included — the site is new).
- Telemetry adds a single aggregate counter, `blueprint_captures`, shared only by
  installations with notice-v3 consent. Blueprint contents, names and hosts are
  never part of telemetry.
