# SpawnWP 0.5.4

This release reshapes the optional **SpawnWP Deploy** plugin around what people
actually reach for — blueprints — and makes getting the plugin onto a cockpit site
a single checkbox.

## The plugin knows where it runs

The plugin used to show three unrelated jobs on one screen: authoring a blueprint,
publishing a site out, and receiving a published site. It now detects its
environment and leads with one thing:

- **Create a blueprint from this site** is the primary panel everywhere. Capture the
  configured plugins, theme, uploads and content as a reusable template on your own
  SpawnWP server.
- **Publishing a finished site elsewhere** moves into a collapsible *Advanced* panel.
  On a cockpit site you get *Send this site to an empty destination*; on an external
  WordPress you get *Receive a published site*. The role that doesn't apply to the
  current site is explained rather than left as a dead button.

The dashboard now wears the SpawnWP mark and wordmark, and shows whether it detected
a **cockpit site** or an **external site** so the available actions make sense at a
glance.

## One checkbox to add it at spawn

Creating a site from the cockpit's **Deploy** page has a new *Install the SpawnWP
Deploy plugin* option. Tick it and the new site comes up with the plugin installed
and activated, ready to create blueprints or publish itself out. Leave it unticked
for a clean site — you can always add the plugin later.

The plugin is signed and **bundled inside the SpawnWP release**, so the cockpit
installs it offline from a verified copy; nothing is fetched from the internet.

## Under the hood

- New `POST /api/new-project` field `install_deploy_plugin`; `new-project.sh`
  installs the bundled, signed plugin zip on the new site when it is set.
- The release build embeds a deterministic copy of the plugin
  (`assets/spawnwp-deploy.zip`), covered by the release signature.
- No migrations; blueprint formats, ingest API and signing model are unchanged.
