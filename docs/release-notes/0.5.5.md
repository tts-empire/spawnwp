# SpawnWP 0.5.5

A focused bug-fix release.

## Content-blueprint sites no longer break on a plugin notice

Spawning a site from a **captured content blueprint** could produce a broken page:
a plugin registered on the source site emitted a WordPress notice during load, and
because debug output was being displayed, the notice was printed before headers were
sent — the site then failed with `Cannot modify header information - headers already
sent`.

Two things caused it, both fixed:

- The container passed `WORDPRESS_DEBUG` as the string `"false"` when debug was off.
  The WordPress image reads that with `!!`, and `!!"false"` is **true**, so `WP_DEBUG`
  was silently switched **on**. `WORDPRESS_DEBUG` is now left empty when off, so
  `WP_DEBUG` is genuinely off for content, clean and demo blueprints (only the
  development blueprint turns it on).
- Debug display was left at WordPress's default (on). Sites now define
  `WP_DEBUG_LOG` (logs to `wp-content/debug.log`) and let on-screen display follow the
  Deploy page's *display_errors* toggle, so a stray plugin notice can never break the
  page unless you explicitly opted into on-screen errors.

## If you already have a broken site

The fix applies to sites created after updating. An already-broken site can be
repaired in place:

```bash
cd /srv/<site>
docker compose exec -T -u www-data php wp config set WP_DEBUG false --raw
```

No migrations; only the per-site Compose template changed.
