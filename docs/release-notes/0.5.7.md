# SpawnWP 0.5.7

A security-hardening release. A source-level audit of the cockpit and the SpawnWP Deploy
plugin found no critical or high-severity issues — the existing defenses (passkeys, TOTP,
hashed sessions, CSRF, signed machine requests, zip hardening, empty-target guards) held up.
This release tightens the remaining low-severity, defense-in-depth edges so that **fresh
installs are born with them in place** and existing installs pick them up on update.

## Cockpit

- **Content-Security-Policy on every page.** The cockpit now sends a strict same-origin CSP
  (`default-src 'self'`, `object-src 'none'`, `base-uri 'none'`, `frame-ancestors 'none'`) on
  all responses, not just the login page — an extra wall against script injection and framing.
- **Stricter output escaping.** The dashboard's HTML escaper now also escapes single quotes and
  backticks, closing a latent gap for values rendered inside inline handlers.
- **Safer Adminer sign-in bridge.** The one-click "open database" page now escapes any markup in
  the injected configuration, so a credential can never break out of the page's script.

## Deploy plugin

- **SSRF-guarded outbound requests.** Pairing and publishing now use WordPress's
  `wp_safe_remote_*` HTTP API, which enforces WordPress's external-host protections.
- **Robust key storage.** The plugin encrypts its connection keys with a key derived from the
  WordPress salts; on a misconfigured install with no salts it now derives from a random secret
  generated and stored once, instead of a predictable value. Installs with normal salts are
  unaffected and need no migration.

No configuration changes and no migrations. Existing paired connections and blueprints keep
working.
