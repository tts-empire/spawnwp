# SpawnWP 0.5.8

Captured content blueprints now **reproduce the origin's WordPress version**. Previously a
blueprint always installed the latest WordPress; now the capture records the source site's
exact core version (e.g. `6.5.2`) and sites spawned from it start on that release.

## Mirror the source WordPress version

- **Captured version is pinned.** The SpawnWP Deploy plugin records the source site's exact
  WordPress version when it captures a blueprint, instead of always `latest`. Beta/RC
  suffixes are reduced to the numeric core; an unreadable version falls back to `latest`.
- **Override at spawn.** On the cockpit's **Deploy** page a captured blueprint offers its
  pinned version *(as captured)* by default, or **Latest** — mirroring how the PHP version is
  already chosen. A soft, non-blocking notice appears if a much older WordPress is paired with
  a modern PHP.
- **Any version, on the fly.** The exact core is fetched from wordpress.org during the image
  build, so any published version works. The base image is now keyed on the PHP version only
  (`wordpress:php<version>-fpm`), which removes the previous coupling between the WordPress
  version and the available image tags.

Pinning an exact version freezes core security updates for that blueprint; apply them inside
the spawned site as usual. Built-in blueprints are unchanged and continue to track `latest`.

## Notes

- No migration. Existing blueprints and sites keep working; blueprints captured before this
  release simply continue to use `latest`.
- The bundled SpawnWP Deploy plugin moves to `0.3.2-dev`.
