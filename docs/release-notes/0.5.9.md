# SpawnWP 0.5.9

This release adds a **per-site file browser** to the cockpit. Until now, inspecting or
editing the files that live outside WordPress's media library and editor — `wp-config.php`,
`.htaccess`, mu-plugins, drop-ins, a plugin's `.env`, a debug log — meant opening an SSH
session. The **📂 Files** button on each site card closes that gap.

## Browse, edit and transfer a site's files

- **Rooted at the document root.** The browser opens at the WordPress docroot
  (`/var/www/html`), covering both WordPress core and `wp-content`. Directories are listed
  first, with size and modification time; the breadcrumb walks back up.
- **View, edit, download, upload.** Click a text file to edit it inline and save it back in
  place; large or binary files are offered as a download instead. Upload files into the
  current folder, create subfolders, and rename or delete entries.
- **Jailed in the container.** Every operation runs inside that one site's PHP container as
  the web user (`www-data`). The container boundary *is* the jail: a path can never escape
  to the host or another site, and written files get the ownership WordPress needs.
- **Read-only by default, writes step up.** Browsing, viewing and downloading are always
  available. The write actions — save, upload, delete, rename, new folder — are treated as
  sensitive and prompt for a recent Passkey confirmation, exactly like Destroy and Restore.

## Notes

- Editing a file in the docroot changes the live site, and writing PHP where the server
  executes it runs that code — by design, it is your site. Take a **💾 Snapshot** before a
  risky edit; a disposable site is always one snapshot (or re-spawn) away from clean.
- No new services or ports. The browser reuses the existing `docker compose exec` path that
  already powers the WP-CLI console, so there is nothing extra to install or expose.
